# 🎯 Playwright Actions (Incremental, Enterprise-Level)

---

## 1. WHAT

**Actions** are the operations Playwright performs on page elements.

These actions simulate real user behavior such as:

* click
* fill
* clear
* press keyboard keys
* select dropdown values
* navigate through business flow

In our SauceDemo project, actions are used to:

* login
* add products
* open cart
* proceed to checkout
* complete order

---

## 2. WHY

Without actions:

* test cannot interact with UI
* no business flow can be automated
* no meaningful end-to-end testing happens

With actions:

* user behavior is simulated
* business flows become executable
* tests become closer to real-world usage
* reusable page methods can be created

---

## 3. WHEN TO USE

Use actions when:

* entering login credentials
* clicking buttons
* adding products to cart
* proceeding to checkout
* completing a form
* triggering keyboard interaction

---

## 4. HOW

Execution flow:

```text
Locator → Action → Page State Change → Assertion
```

SauceDemo example:

```text
Locate Username → Fill Username
Locate Password → Fill Password
Locate Login Button → Click
Inventory Page Loads → Validate
```

---

## 5. REAL-LIFE ANALOGY

Think of actions as your **hands operating the application**.

* locator = finding the object
* action = using the object
* assertion = checking whether the expected result happened

Example:

* find login button
* click login button
* verify inventory page opened

So:

```text
Locator finds
Action performs
Assertion confirms
```

---

## 6. ENGINEERING VIEW

In our project:

* **Locators** identify elements
* **Actions** perform interactions
* **Assertions** validate outcomes

Good automation design means:

* tests should not directly perform all raw actions
* page classes should hold reusable action methods
* tests should read like business flow

So instead of this in test:

```ts
await page.fill('#user-name', 'standard_user');
await page.fill('#password', 'secret_sauce');
await page.click('#login-button');
```

We should use:

```ts
await loginPage.login('standard_user', 'secret_sauce');
```

This is more readable and reusable.

# 7. ACTION TYPES USED IN THIS PROJECT

---

## Fill

Used for:

* username
* password
* first name
* last name
* postal code

Example:

```ts
await this.usernameInput.fill(username);
```

---

## Clear

Used before entering fresh values.

Example:

```ts
await this.usernameInput.clear();
```

---

## Click

Used for:

* login button
* add to cart
* cart icon
* checkout button
* continue button
* finish button

Example:

```ts
await this.loginButton.click();
```

---

## Keyboard Press

Can be used for advanced interaction if required.

Example:

```ts
await this.page.keyboard.press('Tab');
```

---

## Composite Action

A composite action combines multiple smaller actions into one reusable business step.

Example:

```text
Fill checkout form → click continue → click finish
```

This can become:

```ts
await checkoutPage.completeCheckoutFlow(...)
```

## Updated `pages/LoginPage.ts`

```ts
import { Page, expect } from '@playwright/test';

export class LoginPage {
  constructor(private page: Page) {}

  usernameInput = this.page.getByPlaceholder('Username');
  passwordInput = this.page.getByPlaceholder('Password');
  loginButton = this.page.getByRole('button', { name: 'Login' });
  errorMessage = this.page.getByTestId('error');

  async goto() {
    await this.page.goto('https://www.saucedemo.com/');
  }

  async login(username: string, password: string) {
    await this.usernameInput.fill(username);
    await this.passwordInput.fill(password);
    await this.loginButton.click();
  }

  async clearAndLogin(username: string, password: string) {
    await this.usernameInput.clear();
    await this.passwordInput.clear();

    await this.usernameInput.fill(username);
    await this.passwordInput.fill(password);

    await this.loginButton.click();
  }

  async loginUsingKeyboard(username: string, password: string) {
    await this.usernameInput.fill(username);
    await this.page.keyboard.press('Tab');

    await this.passwordInput.fill(password);
    await this.page.keyboard.press('Tab');

    await this.page.keyboard.press('Enter');
  }

  async validateLoginPageVisible() {
    await expect(this.loginButton).toBeVisible();
  }

  async validateLoginErrorVisible() {
    await expect(this.errorMessage).toBeVisible();
  }
}
```

---

## Why this increment?

This adds:

* `clearAndLogin()` for reusable reset + login action
* `loginUsingKeyboard()` for keyboard-driven interaction
* stronger business action abstraction

import { Page, expect } from '@playwright/test';

export class InventoryPage {
  constructor(private page: Page) {}

  backpackAddButton = this.page.locator('#add-to-cart-sauce-labs-backpack');
  bikeLightAddButton = this.page.locator('#add-to-cart-sauce-labs-bike-light');
  cartLink = this.page.locator('.shopping_cart_link');
  title = this.page.getByText('Products');

  async validateInventoryPageLoaded() {
    await expect(this.page).toHaveURL(/inventory/);
    await expect(this.title).toHaveText('Products');
  }

  async addBackpackToCart() {
    await this.backpackAddButton.click();
  }

  async addBikeLightToCart() {
    await this.bikeLightAddButton.click();
  }

  async addProductByName(productName: string) {
    const productCard = this.page
      .locator('.inventory_item')
      .filter({ hasText: productName });

    await productCard.getByRole('button', { name: 'Add to cart' }).click();
  }

  async openCart() {
    await this.cartLink.click();
  }

  async openCartUsingIcon() {
    await this.cartLink.click();
  }
}
```

import { Page, expect } from '@playwright/test';

export class CheckoutPage {
  constructor(private page: Page) {}

  firstName = this.page.locator('#first-name');
  lastName = this.page.locator('#last-name');
  postalCode = this.page.locator('#postal-code');
  continueButton = this.page.getByRole('button', { name: 'Continue' });
  finishButton = this.page.getByRole('button', { name: 'Finish' });
  successHeader = this.page.locator('.complete-header');

  async fillCheckoutInfo(first: string, last: string, zip: string) {
    await this.firstName.fill(first);
    await this.lastName.fill(last);
    await this.postalCode.fill(zip);
  }

  async continueCheckout() {
    await this.continueButton.click();
  }

  async finishCheckout() {
    await this.finishButton.click();
  }

  async completeCheckoutFlow(first: string, last: string, zip: string) {
    await this.fillCheckoutInfo(first, last, zip);
    await this.continueCheckout();
    await this.finishCheckout();
  }

  async validateOrderSuccess() {
    await expect(this.successHeader).toHaveText('Thank you for your order!');
  }
}
```

---

## Why this increment?

This adds:

* `completeCheckoutFlow()` which converts multiple low-level actions into one reusable business action

    inventoryPage = new InventoryPage(page);

    await loginPage.validateLoginPageVisible();
  });

  test('valid login should open inventory page', async () => {
    await loginPage.login(USERS.standard.username, USERS.standard.password);
    await inventoryPage.validateInventoryPageLoaded();
  });

  test('valid login using keyboard should open inventory page', async () => {
    await loginPage.loginUsingKeyboard(
      USERS.standard.username,
      USERS.standard.password
    );
    await inventoryPage.validateInventoryPageLoaded();
  });

  test('locked user should not login successfully', async () => {
    await loginPage.clearAndLogin(
      USERS.locked.username,
      USERS.locked.password
    );
    await loginPage.validateLoginErrorVisible();
  });
});

# 12. COMMON MISTAKES

---

## Mistake 1: Keeping all raw actions in test file

Problem:

* tests become long
* hard to reuse
* difficult to maintain

Fix:

* move action logic into page methods

---

## Mistake 2: Creating action methods without business meaning

Problem:

* code is still too low-level

Fix:

Prefer:

```ts
completeCheckoutFlow()
```

Instead of only:

```ts
fillFirstName()
fillLastName()
fillZip()
clickContinue()
clickFinish()
```

when business grouping is logical.

---

## Mistake 3: Hardcoding only one product action

Problem:

* framework becomes rigid

Fix:

* add dynamic methods like `addProductByName()`

---

## Mistake 4: Using waits instead of action design

Problem:

* flaky tests

Fix:

* rely on Playwright auto-waiting
* keep locators strong
* create meaningful reusable actions

# 13. INTERVIEW NOTES

### What are actions in Playwright?

Actions are user-like interactions such as click, fill, clear, press, and select.

### Why move actions into page classes?

To improve:

* reusability
* readability
* maintainability

### What is a composite action?

A composite action combines multiple smaller UI actions into one business-level reusable method.

Example:

```ts
completeCheckoutFlow()
```

### Why are reusable actions important?

Because enterprise frameworks should describe business flow, not only raw UI steps.
