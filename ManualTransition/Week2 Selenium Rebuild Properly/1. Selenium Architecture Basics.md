# 2.1 Selenium Architecture Basics

## 🎯 What

Selenium architecture defines how automation scripts communicate with browsers.

Core pieces:

* WebDriver model
* Browser drivers
* Session lifecycle

Without architecture understanding:

* tools feel black box
* debugging becomes hard
* framework design stays weak

---

## ❓ Why This Matters in Automation

Every Selenium test uses this internally.

When you run:

```java id="4u0rjr"
driver.get("https://site.com");
```

Many things happen underneath.

Understanding architecture helps with:

* debugging failures
* driver issues
* session problems
* interview discussions

---

## ⏰ When You Use This

Use this understanding when:

* launching browsers
* debugging failures
* handling driver mismatch
* managing sessions

# 🏗 WebDriver Model

WebDriver is interface for browser automation.

Example:

```java id="rqlh4r"
WebDriver driver=
new ChromeDriver();
```

Use:

* send commands
* control browser
* automate actions

---

## Commands Through WebDriver

```java id="8zv0pa"
driver.get(url);

driver.findElement(...);

driver.quit();
```

Commands go through WebDriver model.

---

# Browser Drivers

Drivers translate commands to browser.

Examples:

* ChromeDriver for Google Chrome
* GeckoDriver for Mozilla Firefox
* EdgeDriver for Microsoft Edge

Example:

```java id="3l6dno"
WebDriver driver=
new ChromeDriver();
```

Common mistake:

Browser and driver version mismatch.

---

# Session Lifecycle

Each test starts browser session.

Lifecycle:

```text id="x7m2be"
Create Session
↓
Run Commands
↓
Close Session
```

---

## Start Session

```java id="x43lbm"
new ChromeDriver();
```

Creates unique session.

---

## Execute Session

```java id="exy5yr"
driver.get(url);
```

Runs actions inside session.

---

## End Session

```java id="4zc6ps"
driver.quit();
```

Ends session.

Common mistake:

Forgetting quit().
